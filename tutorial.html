Czym jest Git?

System Kontroli Wersji

Git to rozproszony system kontroli wersji stworzony przez Linusa Torvaldsa (twórcę Linuxa). Pozwala śledzić zmiany w plikach i współpracować z zespołem nad jednym projektem.

Kluczowe cechy Git:

  • • Rozproszona architektura - każdy ma pełną kopię
  • • Szybkość i wydajność
  • • Możliwość pracy offline
  • • Elastyczne modele rozgałęziania

Jak działa Git?

Git przechowuje zdjęcia (snapshots) Twojego projektu w czasie. Za każdym razem, gdy zatwierdzasz zmiany, Git zapisuje migawkę tego, jak wyglądają Twoje pliki w tym momencie.

# Git przechowuje historię zmian
commit 1: Initial project setup
commit 2: Added login functionality
commit 3: Fixed navigation bug
commit 4: Updated styling

Instalacja i konfiguracja

Instalacja Git

Windows:

1. Pobierz z: https://git-scm.com/download/win
2. Uruchom instalator i wybierz domyślne opcje
3. Sprawdź wiersz poleceń: git --version

macOS:

# Używając Homebrew
$ brew install git
# Lub pobierz z git-scm.com
$ git --version

Linux (Ubuntu/Debian):

# Aktualizuj repozytoria
$ sudo apt update
# Instaluj Git
$ sudo apt install git

Podstawowa konfiguracja

Po instalacji Git wymaga podstawowej konfiguracji - Twojego imienia i emaila. Te informacje będą przypisywane do Twoich commitów.

# Konfiguracja globalna (dla wszystkich projektów)
$ git config --global user.name "Twoje Imię i Nazwisko"
$ git config --global user.email "twoj.email@example.com"
# Sprawdź konfigurację
$ git config --list

Przydatne aliasy

Utwórz skróty do często używanych komend:

# Skrócone komendy
git config --global alias.co checkout
git config --global alias.br branch
git config --global alias.ci commit
git config --global alias.st status
# Przydatne aliasy
git config --global alias.unstage 'reset HEAD --'
git config --global alias.last 'log -1 HEAD'
git config --global alias.visual '!gitk'

Weryfikacja instalacji

Sprawdź czy Git jest poprawnie zainstalowany:

# Sprawdź wersję
$ git --version
git version 2.34.1
✓ Git jest gotowy do użycia!

Podstawy Git

Inicjalizacja repozytorium

Rozpocznij śledzenie zmian w swoim projekcie:

# Utwórz nowy folder projektu
$ mkdir moj-projekt
$ cd moj-projekt
# Zainicjalizuj repozytorium Git
$ git init
Initialized empty Git repository in /path/to/moj-projekt/.git/

Cykl życia plików w Git

Git ma trzy główne stany plików: Working Directory, Staging Area i Repository.

Working Directory

Pliki, nad którymi aktualnie pracujesz

Staging Area

Przygotowane zmiany do zatwierdzenia

Repository

Zatwierdzone zmiany w historii

Podstawowe komendy

git status - sprawdź status

$ git status
On branch main
No commits yet
nothing to commit

git add - dodaj pliki

# Dodaj pojedynczy plik
$ git add index.html
# Dodaj wszystkie pliki
$ git add .
# Dodaj wszystkie pliki HTML
$ git add *.html

git commit - zatwierdź zmiany

# Zatwierdź ze skróconym komunikatem
$ git commit -m "Initial commit"
# Zatwierdź z szczegółowym opisem
$ git commit -m "Add homepage" -m "Created basic HTML structure"

Praktyczne ćwiczenie

Wypróbuj podstawowe komendy w praktyce:

# 1. Utwórz plik
$ echo "# Mój Projekt" > README.md
# 2. Sprawdź status
$ git status
# 3. Dodaj plik
$ git add README.md
# 4. Zatwierdź zmiany
$ git commit -m "Add README file"

Przydatne skróty

git add -A Dodaj wszystko
git add -u Dodaj zmodyfikowane
git commit -am Add + commit
git log --oneline Skrócony log

Praca z repozytoriami zdalnymi

Dodawanie i zarządzanie remote

Remote to zdalne repozytorium (np. GitHub). Najczęściej używanym remote jest origin.

# Dodaj remote origin
$ git remote add origin https://github.com/user/repo.git
# Pokaż skonfigurowane remote
$ git remote -v
# Zmień URL
$ git remote set-url origin git@github.com:user/repo.git
# Usuń remote
$ git remote remove origin

Clone, Fetch, Pull, Push

Klonowanie

$ git clone https://github.com/user/repo.git
$ git clone git@github.com:user/repo.git

Pobieranie zmian

$ git fetch origin
$ git pull origin main
$ git pull --rebase origin main

Wysyłanie zmian

# Pierwszy push i ustawienie upstream
$ git push -u origin main
# Kolejne push-e
$ git push

Śledzenie branchy i upstream

Branch śledzący ma połączenie z gałęzią na zdalnym repozytorium.

$ git branch -vv
$ git branch --set-upstream-to=origin/main main
$ git push --set-upstream origin feature/login

Tagi i wydania

# Lekki tag
$ git tag v1.0.0
# Adnotowany tag (zalecany)
$ git tag -a v1.0.0 -m "Pierwsze wydanie"
# Wypchnij tagi
$ git push origin --tags

Ćwiczenie: pierwsze połączenie

$ echo "# Projekt" > README.md
$ git init && git add . && git commit -m "Init"
$ git remote add origin https://github.com/user/repo.git
$ git push -u origin main

Najczęstsze problemy

  • • src refspec main does not match any — brak commitów lub zła gałąź
  • • Konflikty po git pull — rozwiąż konflikty i zatwierdź
  • • SSH Permission denied (publickey) — skonfiguruj klucze SSH

Gałęzie i merge

Tworzenie i przełączanie gałęzi

# Utwórz nową gałąź
$ git branch feature/login
# Przełącz się
$ git checkout feature/login
$ git switch feature/login
# Utwórz i przełącz jednocześnie
$ git checkout -b feature/login
$ git switch -c feature/login

Scalanie (merge) vs Przepisanie historii (rebase)

Merge

$ git checkout main
$ git merge feature/login

Zachowuje historię rozgałęzień. Bezpieczny dla zespołów.

Rebase

$ git checkout feature/login
$ git rebase main

Przepisuje historię, tworzy liniową oś czasu. Nie używaj na publicznych branchach.

Rozwiązywanie konfliktów

$ git merge feature/login
# CONFLICT (content): ...
# 1. Sprawdź status
$ git status
# 2. Otwórz plik i rozwiąż konflikty
<<<<<<< HEAD
Twoja wersja
=======
Wersja z feature/login
>>>>>>> feature/login
# 3. Oznacz jako rozwiązane i zatwierdź
$ git add .
$ git commit -m "Resolve merge conflicts"

Strategie pracy z gałęziami

  • • Git Flow: main, develop, feature/*, release/*, hotfix/*
  • • Trunk-based development: krótkie gałęzie, szybkie merge/rebase
  • • Pull Requests: code review, testy CI przed scaleniem

Ćwiczenie: feature → main

$ git switch -c feature/ui
$ # zrób zmiany, commit
$ git switch main && git pull
$ git merge feature/ui
$ git push

Zaawansowane techniki

Stash: tymczasowe odkładanie zmian

$ git stash
$ git stash list
$ git stash pop
$ git stash apply stash@{2}
$ git stash push -m "praca-w-toku" -- include-untracked

Reflog i odzyskiwanie

$ git reflog
$ git checkout -b restore-branch HASH
$ git fsck --lost-found

Reflog śledzi lokalne ruchy HEAD. Użyj do odzyskiwania utraconych commitów.

Reset vs Revert

git reset

$ git reset --soft HEAD~1
$ git reset --mixed HEAD~1
$ git reset --hard HEAD~1

Zmienia wskaźniki i/lub pliki robocze. Nie używaj --hard bez świadomości konsekwencji.

git revert

$ git revert HEAD
$ git revert a1b2c3d

Tworzy nowy commit odwracający zmiany. Bezpieczny na publicznych branchach.

Cherry-pick i interaktywny rebase

$ git cherry-pick a1b2c3d
$ git rebase -i HEAD~5
# pick | reword | squash | drop

Podmoduły (submodules) — podstawy

$ git submodule add https://github.com/user/lib.git libs/lib
$ git submodule update --init --recursive
$ git submodule foreach git pull

Checklist bezpieczeństwa

  • • Nigdy nie rebase’uj publicznych branchy
  • • Używaj --force-with-lease zamiast --force
  • • Rób tagi dla wydań
  • • Używaj PR + CI przed merge do main
